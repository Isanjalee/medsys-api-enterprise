2. Executive Summary

MEDSYS currently provides a healthcare backend MVP that supports authentication, patient management, families, appointments, encounters, prescriptions, dispensing, inventory, analytics overview, and audit logging. The system is implemented as a TypeScript monorepo with a Fastify API service and a worker service for asynchronous audit persistence.

Business Modules

  • Authentication and session management
  • Patients and families
  • Appointments and encounters
  • Prescriptions and dispensing
  • Inventory and stock movement
  • Analytics overview
  • Audit trail

Operational Characteristics

  • JWT-based access control
  • Organization-scoped data access
  • Transactional clinical write flows
  • Structured logging with PHI redaction
  • Rate limiting
  • OpenAPI documentation
  • Partitioned appointment and audit tables

4. User Roles and Access Model

RolePrimary ResponsibilityCurrent Implemented Access
OwnerAdministrative and supervisory controlCan access all implemented modules within the organization and is the only role that can retrieve audit logs.
DoctorClinical review and care deliveryCan view patient and clinical records, create encounter bundles, add allergies and conditions, view prescriptions, and access analytics. Cannot dispense prescriptions or create families/patients.
AssistantFront-desk and operational supportCan create patients, families, appointments, inventory items, stock movements, dispense prescriptions, and record selected patient support data such as vitals and timeline events.
  • All protected endpoints require a valid access token.
  • The token subject must resolve to an active user record.
  • All business data access is scoped by organization.
  • Role authorization is enforced per route.

6. Functional Requirements

IDAreaRequirement
FR-001AuthenticationThe system shall authenticate users using email, password, and organization ID.
FR-002AuthenticationThe system shall reject invalid credentials and inactive users.
FR-003TokensThe system shall issue RS256 access tokens containing user ID, role, and organization context.
FR-004Refresh TokensThe system shall persist, validate, revoke, and rotate refresh tokens.
FR-005PatientsThe system shall create, list, retrieve, and update patients within the authenticated organization.
FR-006PatientsThe system shall validate NIC format, name format, DOB constraints, and age bounds for patient records.
FR-007PatientsThe system shall provide a combined patient profile containing patient master data, allergies, conditions, recent vitals, and timeline events.
FR-008FamiliesThe system shall create family records and auto-generate a family code when one is not supplied.
FR-009FamiliesThe system shall create family membership links and update the patient master record with the selected family ID.
FR-010AppointmentsThe system shall create, list, retrieve, filter, and update appointments.
FR-011AppointmentsThe system shall support statuses of waiting, in consultation, completed, and cancelled.
FR-012EncountersThe system shall create encounter bundles only from appointments that are in waiting or in consultation state.
FR-013EncountersThe system shall create encounter, diagnoses, tests, optional prescription, optional prescription items, and appointment completion atomically in one transaction.
FR-014EncountersThe system shall enforce a single encounter per appointment.
FR-015PrescriptionsThe system shall list prescriptions and return prescription detail with prescription items and dispense records.
FR-016DispensingThe system shall provide a pending dispense queue for eligible prescriptions without an existing dispense record.
FR-017DispensingThe system shall prevent a prescription from being dispensed more than once.
FR-018DispensingThe system shall verify inventory item existence and sufficient stock before deducting stock during dispense.
FR-019InventoryThe system shall create, list, and update inventory items and movement history.
FR-020InventoryThe system shall reject stock-reducing movements when available stock is insufficient.
FR-021AnalyticsThe system shall provide an analytics overview with counts for patients, waiting appointments, prescriptions, and low-stock inventory items.
FR-022AuditThe system shall write audit records for implemented administrative and clinical actions and allow owners to retrieve them with filters.
FR-023DocumentationThe system shall expose OpenAPI UI at /api/v1/docs and redirect /docs to it.
FR-024HealthThe system shall expose a health check endpoint returning { "ok": true }.
FR-025Audit WorkerThe worker service shall consume Redis-backed audit queue messages and persist them to the audit log table when queue mode is enabled.

7. Non-Functional Requirements

IDCategoryRequirement
NFR-001SecurityThe system shall use RS256 JWT-based authentication for protected API access.
NFR-002SecurityThe system shall require valid RSA key material through environment variables before startup.
NFR-003SecurityThe system shall enforce route-level role-based access control and active-user verification.
NFR-004SecurityThe system shall redact selected credential and PHI fields from structured logs.
NFR-005Rate LimitingThe system shall limit login requests to five per minute and default authenticated requests to one hundred per minute.
NFR-006IntegrityThe system shall preserve transactional integrity for encounter bundle creation and prescription dispensing.
NFR-007IntegrityThe system shall prevent inventory stock from becoming negative through implemented write flows.
NFR-008ReliabilityThe system shall fall back to direct audit persistence when queue publication fails.
NFR-009ReliabilityThe worker service shall shut down cleanly on termination signals.
NFR-010ScalabilityThe system shall support a dedicated analytics/reporting database connection through DATABASE_READ_URL.
NFR-011ScalabilityThe system shall partition appointments by scheduled time and audit logs by creation time.
NFR-012ObservabilityThe system shall provide an API health endpoint and OpenAPI documentation.
NFR-013MaintainabilityThe system shall manage schema evolution through Flyway migrations.
NFR-014MaintainabilityThe system shall validate Flyway migrations in CI using an ephemeral PostgreSQL service.
NFR-015ConfigurationThe system shall fail startup when mandatory runtime environment variables are missing.

9. Key Relationship Summary

  • One organization contains many users, patients, families, appointments, encounters, prescriptions, inventory items, and audit logs.
  • One family can contain many patients through family membership links.
  • One patient can have many appointments, allergies, conditions, vitals, timeline events, encounters, and prescriptions.
  • One appointment can have at most one encounter.
  • One encounter can have many diagnoses and test orders and can produce a prescription.
  • One prescription can have many prescription items and dispense history, although the application permits only a single successful dispense flow.
  • One inventory item can have many movement records.

10. Runtime and Delivery Boundaries

  • Mandatory runtime configuration includes DATABASE_URL, JWT key material, and ORGANIZATION_ID.
  • Optional runtime configuration includes DATABASE_READ_URL for analytics/reporting reads and REDIS_URL for audit queue transport.
  • The application fails startup when mandatory environment variables are missing or invalid.
  • This document reflects the implemented backend only and excludes unimplemented roadmap features such as search, advanced observability, and final infrastructure completion.
This specification is intended to serve as a client-ready description of the current MEDSYS backend implementation and can be used as a baseline delivery artifact.

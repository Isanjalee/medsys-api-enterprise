2. Introduction

MEDSYS is a healthcare backend platform that provides organization-scoped services for patient registration, appointments, consultations, prescriptions, inventory, analytics, auditing, search, and operational security.

2.1 Purpose

This document provides a single source of truth for what the backend currently delivers.

2.2 Aims and Objectives

  • Describe current backend capabilities in business and technical terms.
  • Record implemented functional and non-functional requirements.
  • Summarize the stack, schema, controls, and verification coverage.
  • Provide clients with an auditable delivery baseline.

2.3 Scope

  • API behavior exposed through the implemented /v1/... surface.
  • Worker-based asynchronous audit processing.
  • Implemented schema, tests, and operational controls.

2.4 Out of Scope

  • Commercial terms and support arrangements.
  • Deferred AWS staging rollout and live cloud execution.
  • Frontend implementation details beyond backend-aligned interfaces.
Current delivery status: backend application functionality is implemented through Phases 1, 2, 3, 4, 5, and 7. Phase 6 infrastructure artifacts exist, but AWS rollout is intentionally deferred.

3. Executive Summary

The backend is delivered as a TypeScript monorepo with a Fastify API, a dedicated worker service, PostgreSQL persistence, Flyway migrations, Redis-backed asynchronous audit transport, OpenSearch-backed search with database fallback, and observability plus security hardening controls.

Business Modules

  • Authentication and access control
  • Patients and families
  • Appointments and encounters
  • Prescriptions and dispensing
  • Inventory and analytics
  • Audit, search, and observability

Operational Characteristics

  • Organization-scoped records and permissions
  • JWT access and refresh rotation
  • Partitioned clinical and audit tables
  • Cache invalidation on writes
  • Trace, metrics, and PHI-safe logging
  • CI-backed migration and integration coverage

4.1 Logical Architecture

  • The API exposes versioned REST endpoints for clinical and administrative operations.
  • The worker asynchronously persists audit events and handles retry/DLQ flows.
  • PostgreSQL is the system of record for clinical and administrative transactions.
  • Redis supports messaging, retries, lockout state, and selected cache surfaces.
  • OpenSearch supports patient search acceleration when configured.

9.1 Relationship Summary

  • One organization contains many users, families, patients, appointments, encounters, prescriptions, inventory items, refresh tokens, and audit logs.
  • One family can contain many patients through the family_members join table.
  • One patient can have many allergies, conditions, vitals, timeline events, history entries, appointments, encounters, and prescriptions.
  • One appointment can have at most one encounter.
  • One encounter can have many diagnoses and test orders and can produce one prescription header with multiple prescription items.
  • One inventory item can have many inventory movement records.

11. Runtime and Delivery Boundaries

  • Mandatory runtime variables include DATABASE_URL, ORGANIZATION_ID, and JWT RSA key material.
  • Optional features such as Redis queueing, OpenSearch acceleration, Sentry reporting, and replica reads depend on runtime configuration.
  • Infrastructure-as-code and deployment workflows are implemented, but live AWS execution is deferred and is not claimed here as operational acceptance.

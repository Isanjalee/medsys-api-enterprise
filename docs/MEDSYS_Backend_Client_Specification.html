1. Purpose

This specification gives client teams one stable reference for how to integrate with the MEDSYS backend today. It focuses on implemented behavior rather than future proposals, so frontend and integration teams can build against the live contract without inferring rules from role names or older screenshots.

3. Identity and Authorization Model

Important: the backend no longer expects clients to derive permissions only from the role name. Role and permissions are separate concepts. UI decisions should follow the returned permission list.
ConceptMeaningClient Impact
rolePrimary user identity such as owner, doctor, or assistant.Use for labeling workspace identity and user-management summaries, not as the only authorization source.
permissionsEffective allowed actions for the current user.Use this array to decide which screens, buttons, and flows should be available.
extra_permissionsPer-user permission overrides stored on the user account. These are currently supported for selected doctor accounts only.Useful for admin views and auditability, but UI authorization should still rely on the resolved permissions array.

3.1 Effective Permission Resolution

The backend resolves a user's effective permissions centrally as:

effective permissions = role base permissions + extra_permissions

This means a doctor can stay a doctor while receiving a controlled assistant-support subset of additional actions. The system does not model this as two concurrent roles.

3.2 Assistant-Support Permission Subset

Owners can currently grant the following extra permissions to doctor accounts:

  • patient.write
  • appointment.create
  • family.write
  • inventory.write
  • prescription.dispense

Owner-only permissions such as user administration or audit access are not part of this bundle and should not be expected on doctor accounts.

4.2 Frontend Authorization Rule

Frontend clients should render actions based on the returned permissions array. A doctor with assistant-support permissions must still appear as a doctor in identity displays, but assistant actions should be shown when the required permission is present.

5.1 Example Permission Assignment

{
  "extraPermissions": [
    "patient.write",
    "appointment.create",
    "prescription.dispense"
  ]
}

Only valid permission names are accepted. Unsupported permission names or owner-only permissions are rejected by the backend.

7. Validation and Data Rules Important to Clients

  • Patient first_name and last_name are required. Therefore the stored generated full_name is also never null.
  • Patient dob is required and age is also required. The backend derives and validates age against DOB when provided.
  • Patient nic is optional. The backend now assigns every patient a mandatory patient_code used as the always-available internal identifier.
  • Minor patients without a patient NIC must include guardian information. Clients can either link an existing guardian with guardianPatientId or provide guardian name plus guardian NIC or phone.
  • Guardian-aware patient payload fields supported by create and update flows are familyId, guardianPatientId, guardianName, guardianNic, guardianPhone, and guardianRelationship.
  • User creation supports two request shapes: frontend-friendly name or direct API firstName/lastName.
  • Swagger examples are available on the major create and update endpoints to reduce ambiguity around payload shapes.
  • Permission names are validated. Clients should not send arbitrary strings in extraPermissions.

8. Recommended Frontend Behavior

  1. Load GET /v1/auth/me on session restore and keep the returned permissions list as the UI authorization source.
  2. Use role for display identity and workspace semantics, not for allow-or-deny logic.
  3. Hide or disable create, update, dispense, and inventory mutation actions when the required permission is absent.
  4. Expect a doctor with assistant-support permissions to remain role = doctor.
  5. Use GET /v1/patients for summary lists, GET /v1/search/patients for doctor search by patient/guardian identity, and GET /v1/patients/:id/profile after selection for full demographic and guardian details.
  6. Do not assume patient NIC is always present in list responses or even on the patient record. For minors, use guardian-aware search and treat patient_code as the always-available identifier.
  7. Use Swagger examples as the canonical request-body reference when onboarding client teams.
